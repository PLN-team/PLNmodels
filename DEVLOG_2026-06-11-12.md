## 20. Pas Newton conjoint sur (M, ψ) — remplacement du point fixe (11/06)

**Fichiers** : `src/CovarianceTraits.h`, `src/newton_impl.h`

### Motivation

L'ancienne boucle interne faisait :
1. Step Newton diagonal pour M (avec Armijo)
2. Point fixe exact pour S² : `S² = -1 / (A + diag(Ω))` (pas d'Armijo, convergence immédiate)

Cette séparation ignore le couplage entre M et S (via A = exp(M + S²/2 + O)). La correction de l'un modifie A, qui invalide l'autre. On travaille alternativement sur deux sous-problèmes couplés, ce qui ralentit la convergence.

### Nouveau pas conjoint (M, ψ) avec terme croisé

On paramétrise ψ = log(S²) et on résout simultanément le système 2×2 **par entrée (i,j)** :

```
Hessienne diagonale par blocs (indépendance entre (i,j)) :
  H_MM  = w·(A + ω_jj)
  H_ψψ  = 0.5·w·S²·(A·(1 + 0.5·S²) + ω_jj)
  H_Mψ  = 0.5·w·A·S²                              ← terme croisé (nul si A≈0)

det = H_MM·H_ψψ - H_Mψ²   (clampé à 1e-20)
step_M   = (H_ψψ·grad_M   - H_Mψ·grad_ψ) / det
step_ψ   = (H_MM·grad_ψ   - H_Mψ·grad_M) / det
```

L'Armijo est appliqué conjointement sur `(M - α·Q_step, ψ - α·step_ψ)` avec pente :

```
slope = -accu(grad_M % Q_step) - accu(grad_ψ % step_ψ)
```

où `Q_step = step_M - X * (P_X * step_M)` est la projection hors col(X) pour le B profilé.

### Résultat

Convergence plus robuste sur les grands datasets (microcosm, scRNA). Le terme croisé H_Mψ est faible quand A est petit (régions creuses), fort quand A est grand (signaux forts) — l'approximation diagonale par blocs est exacte dans la limite d'observations indépendantes.

## 21. Investigation et suppression de `block_newton_thresh` (12/06)

**Fichiers** : `src/CovarianceTraits.h`, `src/utils.h`, `src/newton_impl.h`, `src/newton_full_cov.cpp`, `src/newton_fixed_cov.cpp`, `R/utils.R`, `R/PLN.R`

### Origine du paramètre

Avant le pas conjoint, `DenseOmegaImpl` offrait un mode alternatif : pour `p ≤ block_newton_thresh`, on calculait le step exact par **solveur Schur p×p par observation** (complément de Schur sur H_MM, en O(np³)). L'approximation diagonale 2×2 était le fallback pour `p > thresh`.

### Analyse du coût — cas barents (n=89, p=30)

Avec `block_newton_thresh = 30` (défaut), barents full activait le solveur exact :
- Coût : 89 × (920 iters) × (30³ / 6) ≈ **2G FLOPs** par EM step
- Temps observé : 3 s (contre 0.9 s pour nlopt)

Passage à `block_newton_thresh = 0` (approx diagonale) :
- Coût : O(n·p) ≈ 90 × fois moins
- Temps observé : **0.68 s**
- Loglik : **identique** (+14 vs nlopt, inchangé)
- Itérations internes : +15% seulement

La précision supplémentaire du solveur exact n'apporte aucun bénéfice observable, pour un coût prohibitif sur des datasets de taille courante.

### Décision : suppression complète

Le paramètre `block_newton_thresh` est **supprimé** de l'ensemble du code (7 fichiers modifiés) :

| Fichier | Modification |
|---------|-------------|
| `R/utils.R` | Suppression de `block_newton_thresh = 0L` dans `config_default_homemade` |
| `R/PLN.R` | Suppression de la ligne `@param` dans la doc |
| `src/utils.h` | Suppression du champ `int block_newton_thresh = 30` dans `NewtonConfig` et du parsing |
| `src/CovarianceTraits.h` | `DenseOmegaImpl::compute_joint_step_MS` et `compute_step_M` : suppression du paramètre et du bloc `if (thresh > 0)` avec les solveurs Schur |
| `src/CovarianceTraits.h` | `DiagonalCovTraits`, `SphericalCovTraits` : suppression du paramètre commenté `/*block_thresh*/ = 0` |
| `src/newton_impl.h` | `newton_optimize_impl` et `newton_vestep_impl` : suppression du paramètre ; correction de virgule résiduelle |
| `src/newton_full_cov.cpp`, `newton_fixed_cov.cpp` | Suppression de `cfg.block_newton_thresh` dans les appels |

## Récapitulatif des décisions algorithmiques finales

### Ce qui est fixé (stable)

| Composante | Choix retenu | Raison |
|-----------|-------------|--------|
| Paramétrage S | ψ = log(S²) | Évite les contraintes ≥ 0, meilleur conditionnement |
| Step interne | Pas conjoint Newton 2×2 (M, ψ) | Capture le couplage M-S via H_Mψ, sans coût O(p³) |
| Approximation hessienne | Diagonale par blocs (indépendance i,j) | Exacte pour Schur diagonal, O(np), identique en qualité à l'exact |
| Armijo | Conjoint sur (M, ψ), pente Q_step projetée | Correct avec B profilé (sinon pas trop longs si d/n grand) |
| B profilé | P_X = solve(X'WX, X'W), mis à jour avant chaque step | Élimine le cold-start, enveloppe valide |
| Structure EM | Boucle interne VE (ftol=1e-8) + boucle externe EM (maxit=200, em_tol=1e-8) | Convergence robuste sur grand n |
| Ω, B | M-step analytique après convergence interne | Newton pure uniquement sur (M, ψ) |

### Ce qui est abandonné

| Idée | Pourquoi abandonnée |
|------|---------------------|
| Block Newton (solveur Schur p×p) | O(np³) prohibitif, aucun gain en loglik vs approximation diagonale |
| Point fixe séparé pour S² | Ignore le couplage M-S, convergence plus lente |
| `homemade_alt` distinct de `homemade` | Fusionné dans `homemade` après validation (B profilé + Q_step Armijo) |
| `hybrid` comme défaut | Newton seul suffit sur tous les cas validés |

### Cas limites connus

- **mollusk** : mauvaise initialisation, les deux backends convergent mal
- **trichoptera diagonal** : besoin de > 200 EM iterations (budget à augmenter localement si nécessaire)
- **microcosm full** : Newton 4–5× plus lent que nlopt (mais seul à converger correctement)
- **La comparaison de temps est sensible à la charge BLAS** (multithreadé) : ne pas lancer les scripts en parallèle

## 24. ZIPLN — Étape VE conjointe (M, ψ, R) et alignement API (12/06)

**Fichiers** : `src/optim_zi-pln.cpp`, `R/ZIPLNfit-class.R`, `R/ZIPLN.R`, `R/RcppExports.R`, `src/RcppExports.cpp`

### Architecture finale de l'étape VE ZIPLN

Les deux backends gèrent R **en interne** et retournent `(M, S², R)` :

**Backend "homemade" — `ve_step_zipln_newton`**
```
Pour chaque itération :
  R     = σ(A + logit(Pi)) × 1{Y=0}     ← mis à jour à chaque iter depuis (M,S²) courant
  one_m_R = 1 - R
  Hessienne 2×2 par (i,j) :
    h_mm = (1-R)·A + ω_jj
    h_mp = 0.5·S²·(1-R)·A               ← terme croisé (théorème de l'enveloppe)
    h_pp = h_mp·(1 + 0.5·S²) + 0.5·S²·ω_jj
  Pas Newton + Armijo conjoint (M, ψ)
convergence : |f(t)-f(t-1)| < ftol·(1+|f(t-1)|)
```

**Backend "nlopt" — `ve_step_zipln_nlopt`**
```
R₀ = σ(A₀ + logit(Pi)) × 1{Y=0}   ← calculé une fois sur (init_M, init_S²)
R₀ fixé pendant toute la solve nlopt  ← garantit cohérence (objectif, gradient)
Après convergence : R = σ(A_final + logit(Pi)) × 1{Y=0}
Retourne (M, S², R)
```

La fixation de R₀ pendant l'optimisation nlopt est nécessaire : mettre à jour R dans chaque callback de gradient brise la cohérence (objectif, gradient) du point de vue de nlopt, dégradant l'ELBO de ~1000 unités (−36706 vs −33523 sur oaks).

### Alignement API PLN ↔ ZIPLN

`ZIPLN_param()` utilise désormais `make_config_optim()` avec `backend = c("homemade", "nlopt")`, comme `PLN_param()`. Les fonctions renommées :
- `optim_zipln_M_psi`        → `ve_step_zipln_nlopt`
- `optim_zipln_M_psi_newton` → `ve_step_zipln_newton`

### Benchmark (oaks, n=116, p=114, `Abundance ~ 1`)

| Backend | ELBO | Temps |
|---------|------|-------|
| homemade (Newton) | −32 982 | 16.1 s |
| nlopt (CCSAQ)     | −33 517 | 14.2 s |

Newton trouve un ELBO meilleur (+535), à temps comparable.

## 26. Investigation et suppression de `approx_ZI` (12/06)

**Fichiers** : `R/ZIPLNfit-class.R`, `R/ZIPLN.R`, `R/ZIPLNnetwork.R`, `tests/testthat/test-zipln.R`

### Contexte

L'option `approx_ZI` existait depuis l'ancienne architecture (R séparé du step (M,S)) pour basculer entre deux formes de mise à jour de R :
- `approx_ZI=TRUE` → `optim_zipln_R_var` : R* = σ(A + logit(π)), maximiseur exact de l'ELBO w.r.t. R
- `approx_ZI=FALSE` → `optim_zipln_R_exact` : R = π / (Φ(O+XB, σ²)·(1-π) + π), vraie probabilité bayésienne P(C=1|Y=0) marginalisée sur Z via la fonction W de Lambert

Dans la nouvelle architecture conjointe, `approx_ZI` n'était plus câblé — le step VE utilisait toujours la forme σ, mais le paramètre restait dans le config sans effet.

### Benchmark de comparaison (trichoptera)

| Backend | approx_ZI | ELBO | Itérations |
|---------|-----------|------|-----------|
| Newton  | TRUE  (σ formula) | **−1139.3** | 167 |
| Newton  | FALSE (Lambert W) | −1216.2 | 16 |
| CCSAQ   | TRUE  (σ formula) | **−1140.8** | 64 |
| CCSAQ   | FALSE (Lambert W) | −1226.5 | 27 |

### Analyse

La forme "exacte" bayésienne donne un ELBO inférieur de **77–86 points**. La raison est théoriquement claire : la formule σ(A + logit(π)) est l'**optimum exact de l'ELBO** par rapport à R (∂ELBO/∂R_ij = 0 donne exactement cette formule). La forme Lambert W maximise la vraisemblance marginale vraie, une fonctionnelle différente — convergence plus rapide mais vers un point sous-optimal au sens de l'ELBO.

Le qualificatif "approché" pour la formule σ était trompeur : c'est la forme exacte du point de vue de la borne variationnelle.

### Décision : suppression complète

`approx_ZI` supprimé de :
- `R/ZIPLN.R` : doc + `make_config_optim(extra=...)`
- `R/ZIPLNnetwork.R` : ligne `config_opt$approx_ZI <- TRUE`
- `tests/testthat/test-zipln.R` : test entier supprimé (testait l'équivalence des deux formes, fausse)

Les fonctions C++ `optim_zipln_R_var` et `optim_zipln_R_exact` sont conservées dans `src/optim_zi-pln.cpp` pour mémoire historique.

## 27. PLNPCA — Remplacement du gradient spectral par L-BFGS joint + Wolfe (12/06)

**Fichier** : `src/builtin_rank_cov.cpp` (ex `spectral_rank_cov.cpp`)

### Historique des tentatives

| Approche | Résultat |
|----------|---------|
| Gradient spectral BB+GLL | Converge vers un mauvais minimum local sur oaks |
| Armijo monotone (nm_window=1) | Identique — problème structural, pas ligne de recherche |
| EM alterné BB (blocs VE / M) | Même minimum — BB diagonal contaminé par couplage M·Cᵀ |
| L-BFGS alterné par blocs + Armijo | Histoire toujours vide (Armijo ne garantit pas s^T y > 0) |
| **L-BFGS joint + Wolfe fort** | **Succès — meilleur loglik que nlopt sur tous les datasets** |
| L-BFGS par blocs EM + Wolfe interne | Pas mieux que joint, 7–10× plus lent |

### Architecture retenue

Optimisation jointe de tous les paramètres `[vec(B); vec(C); vec(M); vec(ψ)]` avec :
- **L-BFGS** deux-boucles (m=10 paires), initialisation steepest-descent
- **Recherche de Wolfe forte** (Nocedal & Wright Algo 3.5/3.6, c₁=1e-4, c₂=0.9)
  - Garantit s^T y > 0 à chaque pas accepté → historique toujours valide
  - Capture la courbure croisée M·Cᵀ que les méthodes bloc-alternantes manquent
- Warm-start ψ : un pas de point fixe `ψ = −log(1 + A·C²)` avant le premier appel L-BFGS
- Convergence : ftol consécutif + fenêtre glissante de 100 itérations

### Benchmark (séquentiel)

| Dataset / rank | nlopt (CCSAQ) | L-BFGS joint |
|---|---|---|
| trichoptera q=3 | −766.9 / 0.68s | **−763.7** / 0.9s |
| trichoptera q=4 | −700.6 / 0.54s | **−693.6** / 2.5s |
| trichoptera q=5 | −686.2 / 0.70s | **−672.6** / 2.9s |
| oaks q=5 | −97677 / 14.6s | **−78226** / 8.6s† |
| oaks q=10 | −66399 / 15.9s | **−47238** / 17s |

†Résultat de session précédente (joint L-BFGS pur) : −77058 / 8.6s. L-BFGS systématiquement meilleur en loglik, couplage M·Cᵀ bien capturé.

### Pourquoi le bloc-EM L-BFGS est-il moins bon ?

Chaque bloc (VE : M,ψ) et (M-step : B,C) est convexe isolément → L-BFGS converge vite en interne. Mais la convergence EM extérieure est linéaire et annule ce gain : 7–10× plus lent et loglik légèrement pire que joint sur oaks.

## 31. Correction de l'initialisation de M pour PLNPCA (12/06)

**Fichier** : `R/PLNPCAfit-class.R` (ligne 262)

### Bug

L'initialisation de `private$M` était :
```r
M <- svdM$u[, 1:q] %*% diag(d[1:q]) %*% t(svdM$v[1:q, 1:q])
```

`svdM$v[1:q, 1:q]` prend les premières `q` **lignes** (espèces) de V, pas les premières `q` composantes. C'est une erreur d'index qui donne un résultat numériquement arbitraire.

### Correction

Dans PLNPCA, le modèle est `Z = O + X·B + M·Cᵀ` avec `M` de taille n×q. La contrainte cohérente avec `C = V[:,1:q]·D[1:q]/√n` est que `M·Cᵀ ≈ M_PLN`. En résolvant :

M · (V[:,1:q]·D[1:q]/√n)ᵀ = U·D·Vᵀ  ⟹  M = √n · U[:,1:q]

```r
private$M <- sqrt(self$n) * svdM$u[, 1:rank, drop = FALSE]
```

### Impact mesuré (trichoptera)

| q | Erreur ‖M·Cᵀ − M_PLN‖_F (ancienne) | Erreur ‖M·Cᵀ − M_PLN‖_F (nouvelle) |
|---|---|---|
| 3 | **1174** | **13.6** |
| 5 | **1176** | **8.4** |

### Validation

72 tests `test-plnpcafit.R` passent. Loglik post-convergence inchangés (l'optimiseur L-BFGS corrigeait de toute façon, mais le point de départ est maintenant cohérent).

## 32. PLNPCA — Retour à nlopt comme backend par défaut (12/06)

**Fichiers** : `R/PLNPCA.R`, `src/builtin_optim_plnpca.cpp`

## 35. PLNPCA — `init_method = "EM"` : EM tronqué pour l'initialisation SVD (15/06)

**Fichiers** : `R/PLNPCA.R`, `R/PLNPCAfamily-class.R`

### Motivation

L'init par défaut `"LM"` donne `M₀ = log((1+Y)/exp(O))` — une transformation log naïve, jamais mise à jour par le VE-step. Pour les grands rangs (`rank > sqrt(p)`), l'écart entre M₀ et la vraie moyenne latente se traduit par un mauvais SVD résiduel et une initialisation sous-optimale de C.

L'alternative `inception = PLN(...)` donne la meilleure qualité mais exige la convergence complète du PLN (coûteux). L'idée : **EM tronqué** — K itérations EM seulement, avec le backend builtin (Newton, rapide), pour obtenir une M plus proche de la vraie valeur sans payer le coût complet.

### Implémentation

Ajout de `init_method = "EM"` et `init_niter = 5L` dans `PLNPCA_param()`.

Dans `PLNPCAfamily$initialize`, nouveau branchement :

```r
} else if (identical(control$init_method, "EM")) {
  niter  <- if (is.null(control$init_niter)) 20L else control$init_niter
  em_ctrl <- PLN_param(backend = "builtin", trace = 0,
                       config_optim = list(maxit_em = niter))
  pln_em <- PLNfit$new(responses, covariates, offsets, weights, formula, em_ctrl)
  pln_em$optimize(responses, covariates, offsets, weights, em_ctrl$config_optim)
  init_B <- pln_em$model_par$B
  init_M <- pln_em$var_par$M
```

**Subtilité** : après calcul du SVD, `control$init_method` est forcé à `"LM"` avant la création des `PLNPCAfit` individuels — sinon `PLNfit$initialize` (appelé par `super$initialize`) passerait `"EM"` à `compute_PLN_starting_point` qui ne le reconnaît pas.

### Résultats — oaks (p=114), scan `init_niter`

| init | rank=5 | rank=20 |
|------|--------|---------|
| LM (défaut) | **−77189.8** | −30654.7 |
| EM K=2 | −77444 | **−30525.6** (+129) |
| EM K=5 | −78003 | **−30518.8** (+135) |
| EM K=10 | −78437 | −30547.5 (+107) |
| EM K=20 | −78443 | −30620.5 (+34) |

### Leçon clé — phénomène de sur-convergence

L'EM tronqué aide uniquement pour **`rank > sqrt(p)`**. Pour les petits rangs, il nuit systématiquement (rank=5, p=114 : −800 ll). La cause : après K itérations, la M s'est "spécialisée" pour expliquer l'espace complet à p dimensions. Les K directions dominantes du SVD de cette M sur-convergée sont moins discriminantes pour un rang faible.

Le sweet spot est **K = 2–5** : au-delà, l'effet de sur-convergence domine. Défaut fixé à `init_niter = 5L`.

### API

```r
# grands rangs (rank > sqrt(p)) :
PLNPCA_param(init_method = "EM")              # K=5 par défaut
PLNPCA_param(init_method = "EM", init_niter = 3)

# qualité maximale (coûteux) :
PLNPCA_param(inception = PLN(formula, data))
```

`"LM"` reste le défaut. `"EM"` est un opt-in documenté.

## 36. PLNnetwork — `inception_backend` et `inception_niter` : init plus rapide et meilleure (15/06)

**Fichiers** : `R/PLNnetwork.R`, `R/PLNnetworkfamily-class.R`

### Motivation

`PLNnetworkfamily$initialize` lance toujours un PLN complet avant la grille de pénalités. Sur oaks (p=114), ce PLN d'inception prend ~4s (nlopt) sur les ~6.5s totales.

Par ailleurs, `inception_cov = "diagonal"` était déjà supporté mais **cassé** : la Σ diagonale donne des off-diagonaux nuls → `max_pen = 0` → grille de pénalités invalide.

### Deux apports

**1. Correctif `inception_cov != "full"`**

Dans `Networkfamily$initialize`, la Σ pour `max_pen` est désormais calculée comme covariance empirique des résidus latents quand la Σ fittée n'est pas plein-rang :

```r
Sigma_maxpen <- if (!is.null(control$sigma_inception)) {
  control$sigma_inception          # résidus empiriques (M − X·B)ᵀ(M − X·B)/n
} else {
  as.matrix(control$inception$model_par$Sigma)
}
```

`sigma_inception` est calculé dans `PLNnetworkfamily$initialize` quand `inception_cov != "full"` ou `inception_niter` est fixé.

**2. `inception_backend` et `inception_niter`**

Deux nouveaux paramètres dans `PLNnetwork_param()` :

- `inception_backend = NULL` : backend pour le PLN d'inception seulement (le reste de la grille utilise `backend`).
- `inception_niter = NULL` : limite les itérations de l'inception (`maxit_em` pour builtin, `maxeval × 10` pour nlopt).

Implémentation : un `ctrl_inc` séparé est construit via `make_config_optim(inc_backend, ...)` et utilisé pour `PLNfit$new()` **et** `PLNfit$optimize()` (le backend doit être cohérent entre les deux car `setup_optimizer` est appelé à `new()`).

### Résultats — benchmark `inception_backend = "builtin", inception_niter = 5`

| Dataset | nlopt full (défaut) | builtin K=5 | Δ ll | speedup |
|---------|-------------------|-------------|------|---------|
| trichoptera (p=17) | −1149 | −1148 | **+1** | ~1× |
| barents (p=30) | −5226 | −5106 | **+120** | ~1× |
| oaks (p=114) | −37827 | −37435 | **+392** | 1.4× |

Le même phénomène de sur-convergence que PLNPCA s'applique : un inception PLN trop convergé (K=50 ou convergence complète builtin) donne un mauvais warm-start pour les modèles pénalisés. Le sweet spot est K=5–10.

### Leçon clé — M sur-convergée nuit au warm-start pénalisé

Un inception complètement convergé fournit une M optimale pour le problème **non-pénalisé**. Or, les modèles de la grille GLASSO ont besoin d'une M qui s'adapte à des Ω contraints (sparse). Partir d'une M très bien ajustée au PLN plein rend la transition vers les solutions sparse plus difficile. L'EM tronqué laisse M "à mi-chemin" — meilleur point de départ pour la relaxation pénalisée.

Ce principe est général : **l'init d'une collection de modèles plus contraints bénéficie d'une initialisation sous-convergée du modèle père**.

### API

```r
PLNnetwork_param(inception_backend = "builtin", inception_niter = 5)
```

Recommandé pour grands p. Les défauts restent inchangés (pas de breaking change).

## 37. PLNnetwork — Warm-start de Ω dans GLASSO : expérience et abandon (15/06)

**Fichiers** : `R/PLNnetworkfit-class.R`, `R/PLNnetworkfamily-class.R` (revertés)

### Hypothèse

`glassoFast` supporte `start = "warm"` avec `w.init` (Σ) et `wi.init` (Ω). L'idée : passer le Ω de l'itération précédente comme point de départ du prochain appel GLASSO.

Deux variantes testées :
1. **Intra-boucle** : warm-start Ω entre les itérations GLASSO→VEM d'un même modèle.
2. **Inter-modèles** : passer `Omega = models[[m]]$model_par$Omega` au modèle m+1 via `update()`.

### Résultats — oaks

| Variante | ll | temps | EM iters |
|----------|-----|-------|---------|
| cold (défaut) | **−37827** | **6.6s** | 90 |
| warm intra-boucle | −37878 | 7.6s | 90 |
| warm inter-modèles | −37838 | 10.9s | 88 |

Toutes les variantes sont pires en qualité **et** en temps.

### Explication

**Intra-boucle** : entre deux appels GLASSO successifs, la Σ change (le VEM met à jour M et B → nouvelle Σ via le M-step). Le Ω optimal sous Σ_{k-1} est loin de l'optimum sous Σ_k. Le cold start (depuis la diagonale de S) est plus proche de la solution sous la nouvelle Σ.

**Inter-modèles** : le path de pénalités est **décroissant** (sparse → dense). Le Ω sparse du modèle m est un mauvais point de départ pour le modèle m+1 moins pénalisé : glassoFast doit "activer" de nouveaux edges depuis une solution trop sparse, ce qui est plus lent que partir de la diagonale. L'algorithme GLASSO warm-start standard fonctionne dans l'autre sens (dense → sparse).

### Conclusion

Le cold start glassoFast (initialisation diagonale interne) est optimal dans les deux contextes du PLNnetwork. **Revert complet.** L'implémentation existante était correcte.

## 40. Uniformisation de la normalisation X pour tous les modèles et backends (16/06)

**Fichiers** : `R/utils.R`, `R/PLNfit-class.R`, `R/PLNnetworkfit-class.R`, `R/ZIPLNfit-class.R`, `R/PLNPCAfit-class.R`

### Motivation

La normalisation des colonnes de X stabilise l'optimisation pour les covariables à grande magnitude (Depth~300). Elle était déjà appliquée dans PLNPCAfit et (après §39) dans `torch_optimize`. Objectif : l'appliquer uniformément à tous les modèles (PLN, PLNPCA, PLNnetwork, ZIPLN) et tous les backends (nlopt, builtin, torch).

### Architecture

**Helper centralisé** dans `utils.R` :
```r
normalize_covariates <- function(X) {
  scales <- pmax(sqrt(colSums(X^2)), 1)
  list(X_sc = sweep(X, 2, scales, "/"), scales = scales)
}
```

**Pattern uniforme** dans chaque `optimize()` :
```r
nrm  <- normalize_covariates(covariates)
# passe X_sc et B_sc = sweep(B, 1, nrm$scales, "*") aux backends
# après optimisation : B <- sweep(B_sc, 1, nrm$scales, "/")
```

**Invariant** : `X_sc %*% B_sc = X %*% B` — le modèle est mathématiquement identique.

### Changements par fichier

| Fichier | Changement |
|---------|------------|
| `R/utils.R` | Ajout de `normalize_covariates()` |
| `R/PLNfit-class.R` | `PLNfit$optimize()` et `PLNfit_fixedcov$optimize()` : normalise X/B, dénormalise B en sortie ; `torch_optimize` : normalisation supprimée (faite en amont) |
| `R/PLNnetworkfit-class.R` | `optimize()` : nrm avant la boucle EM, boucle entière en B_sc (warm-start inclus), dénormalisation après la boucle |
| `R/ZIPLNfit-class.R` | `optimize()` : X local normalisé au début, `parameters$B` initialisé en B_sc, dénormalisé dans `self$update()` |
| `R/PLNPCAfit-class.R` | `optimize()` : code inline remplacé par appel au helper (logique inchangée) |

PLNLDAfit appelle `super$optimize()` → couvert automatiquement, sans modification.

### Résultats numériques

**builtin (Newton 2e ordre) : identique** — Newton H⁻¹g est invariant à l'échelle, résultat mathématiquement exact.

## 41. Migration future/future.apply → parallel/mclapply (16/06)

**Fichiers** : `R/utils.R`, `R/PLNfit-class.R`, `R/PLNPCAfamily-class.R`, `R/PLNnetworkfamily-class.R`, `R/PLNmixturefamily-class.R`, `R/PLNmixture.R`, `R/PLNPCA.R`, `R/PLNmodels-package.R`, `R/PLNPCAfamily-class.R`, `DESCRIPTION`

### Motivation

`future` + `future.apply` interagissent mal avec les bibliothèques BLAS multithreadées (OpenBLAS, MKL) : chaque worker `future::plan("multisession")` spawne un nouveau processus R qui réinitialise BLAS avec N threads → oversubscription (ex. 4 workers × 8 threads BLAS = 32 threads sur 8 cœurs).

`parallel::mclapply` utilise `fork()` — les workers héritent exactement de la config BLAS du parent. L'utilisateur contrôle l'équilibre directement :
```r
Sys.setenv(OPENBLAS_NUM_THREADS = "2")  # 2 threads BLAS par worker
options(mc.cores = 4)                    # 4 workers → 8 cœurs au total
```

## 42. ZIPLN — Remplacement de l'init `pscl::zeroinfl` par LM rapide (16/06)

**Fichiers** : `R/utils.R`, `R/ZIPLNfit-class.R`, `DESCRIPTION`

### Motivation

L'initialisation de ZIPLN parcourait toutes les `p` espèces en boucle et ajustait un modèle ZI Poisson univarié par espèce via `pscl::zeroinfl` (EM complet, 2 GLM par itération). Sur oaks (p=114), cela représentait ~2 s d'init avant même de commencer l'EM ELBO.

Le point de départ n'a pas besoin d'être la solution exacte du modèle ZI Poisson univarié — il suffit d'être dans le bon voisinage pour que l'ELBO converge. De plus, le modèle univarié ignore la structure de covariance jointe, ce qui peut en fait orienter l'init vers un mauvais bassin pour l'ELBO multivarié.

### Nouvelle fonction `compute_ZIPLN_starting_point`

Ajoutée dans `R/utils.R`, exportée publiquement :

```r
compute_ZIPLN_starting_point <- function(Y, X, X0, O, w = NULL) {
  # PLN component: 1 appel lm.fit multivarié (identique à compute_PLN_starting_point "LM")
  sp <- compute_PLN_starting_point(Y, X, O, w, method = "LM")

  # ZI component: taux de zéros empiriques par espèce
  zero_ind <- (Y == 0) * 1.0
  R <- matrix(colMeans(zero_ind), n, p, byrow = TRUE)

  # B0: p GLMs binomiaux sur indicateur de zéro vs X0 (seulement pour ziparam="covar", d0>0)
  B0 <- if (d0 > 0) vapply(1:p, \(j) glm.fit(X0, zero_ind[,j], binomial())$coef, numeric(d0))
        else matrix(0.0, 0L, p)

  list(B = sp$B, M = sp$M, S2 = sp$S2, R = R, B0 = B0)
}
```

### Changements dans `ZIPLNfit$initialize`

La boucle `for (j in 1:p)` avec `pscl::zeroinfl(...)` est remplacée par un unique appel :

```r
sp <- compute_ZIPLN_starting_point(data$Y, data$X, data$X0, data$O, data$w)
private$B <- sp$B; private$B0 <- sp$B0
private$M <- sp$M; private$S2 <- sp$S2; private$R <- sp$R
```

Garde ajoutée pour le cas dégénéré `ncol(X)==0` (préserve l'erreur attendue par les tests).

### Nettoyage

- `@importFrom pscl zeroinfl` et `@importFrom tidyr replace_na` retirés de `ZIPLNfit-class.R`
- `pscl` retiré des `Imports` de `DESCRIPTION` (dépendance lourde, plus nécessaire)
